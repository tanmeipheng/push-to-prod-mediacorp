## Root Cause
HTTP 429 Too Many Requests returned by Partner API endpoint, indicating the client has exceeded the allowed request rate limit.

The upstream API returned transient errors that the current codebase did not handle.
The worker process crashed without any retry or backoff mechanism.

## Remediation Applied
- Added `tenacity` imports: `retry`, `stop_after_attempt`, `wait_exponential`, `retry_if_exception`, `before_sleep_log`, `after_log`.
- Added `logging` setup so tenacity can log retry attempts and sleep intervals.
- Introduced `is_rate_limit_error()` predicate that returns `True` only when the raised `HTTPError` has HTTP status 429, ensuring other HTTP errors are not retried.
- Decorated `sync_data()` with `@retry` using:
  - `retry=retry_if_exception(is_rate_limit_error)` — only retry on 429 errors.
  - `wait=wait_exponential(multiplier=1, min=2, max=60)` — exponential backoff starting at 2 s, capped at 60 s.
  - `stop=stop_after_attempt(6)` — give up after 6 total attempts.
  - `before_sleep` / `after` log hooks for observability.
  - `reraise=True` — re-raises the original exception when retries are exhausted.
- Original function signature and docstring are preserved unchanged.
- Test file covers: immediate success, single 429 retry, multiple 429 retries, exhausted retries raising HTTPError, non-retried 500/404 errors, and correct payload after retry.
