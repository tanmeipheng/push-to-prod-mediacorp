## Root Cause
HTTP 429 Too Many Requests returned by Partner API endpoint, indicating the client has exceeded the allowed request rate limit.

The upstream API returned transient errors that the current codebase did not handle.
The worker process crashed without any retry or backoff mechanism.

## Remediation Applied
- Added `tenacity` imports: `retry`, `stop_after_attempt`, `wait_exponential`, `retry_if_exception`, `before_sleep_log`, `RetryError`.
- Added `logging` setup so tenacity can log retry warnings before each sleep.
- Introduced `is_rate_limit_error(exception)` predicate that returns `True` only for `requests.exceptions.HTTPError` with HTTP status code 429, ensuring other errors are not retried.
- Decorated `sync_data` with `@retry` using:
  - `retry=retry_if_exception(is_rate_limit_error)` — retries only on 429 errors.
  - `wait=wait_exponential(multiplier=1, min=2, max=60)` — exponential backoff starting at 2 s, capped at 60 s.
  - `stop=stop_after_attempt(6)` — maximum of 6 total attempts.
  - `before_sleep=before_sleep_log(logger, logging.WARNING)` — logs each retry attempt.
  - `reraise=True` — re-raises the original `HTTPError` instead of wrapping it in `RetryError`.
- Updated `__main__` block to catch `RetryError` and log a clear message before re-raising.
- Test file covers: immediate success, single retry, multiple retries, exhausted retries, no-retry on 500, no-retry on `ConnectionError`, and correct data returned after retry.
