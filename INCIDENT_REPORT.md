## Root Cause
HTTP 429 Too Many Requests returned by Partner API endpoint, indicating the client has exceeded the allowed request rate limit.

The upstream API returned transient errors that the current codebase did not handle.
The worker process crashed without any retry or backoff mechanism.

## Remediation Applied
• Added `tenacity` imports: `retry`, `retry_if_exception`, `stop_after_attempt`, `wait_exponential`, `before_sleep_log`.
• Added `logging` setup so tenacity can log retry warnings via `before_sleep_log`.
• Introduced `_is_rate_limit_error(exc)` predicate that returns `True` only when the exception is an `HTTPError` with HTTP status 429, ensuring other errors are not retried.
• Decorated `sync_data()` with `@retry` configured for:
  - `retry=retry_if_exception(_is_rate_limit_error)` — retries only on 429 responses.
  - `wait=wait_exponential(multiplier=1, min=2, max=60)` — exponential backoff starting at 2 s, capped at 60 s.
  - `stop=stop_after_attempt(6)` — gives up after 6 total attempts.
  - `reraise=True` — re-raises the original exception after exhausting retries.
• Preserved original function signature, docstring, and all existing logic.
• Test file covers: immediate success, retry-then-success (1 prior 429), multiple retries before success, exhausted retries raising HTTPError, and no retry on non-429 HTTP errors (500, 404).
• Tests patch `tenacity.nap.time` to eliminate real sleep delays during testing.
